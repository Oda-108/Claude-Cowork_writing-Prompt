# なぜClaude Coworkの X 投稿は品質が高いのか — プロンプト設計の解剖

## 結論

品質が高い原因は「Claudeが賢いから」ではない。織田さんが設計した**制約システム**がClaudeの出力を強制的に高品質ゾーンに押し込んでいるからだ。LLMは自由度が高いほど品質がブレる。逆に、正しい制約を与えれば安定して高品質な出力を量産できる。今回のX運用はその最良の実例になっている。

---

## 1. 制約による品質ロック — 自由を奪うことで品質を上げる

### 1-1. 「140文字ぴったり」の強制力

この制約が全体のクオリティを決定づけている。ポイントは「140文字**以内**」ではなく「140文字**ぴったり**」であること。

「以内」だと、Claudeは120文字くらいで止めたり、80文字で書いてしまう場合がある。余白が残り、密度が下がる。「ぴったり」と指定することで、Claudeは以下の思考プロセスを強制される。

1. まず自然に文章を書く（大抵130〜150文字になる）
2. 文字数をカウントする
3. 1〜2文字単位で調整する — 不要な助詞を削る、語尾を変える、より短く強い表現に差し替える

この「1文字単位の推敲プロセス」が、人間のコピーライターが推敲するときとまったく同じ思考を再現している。結果として「無駄な言葉が物理的に存在できない文章」が生まれる。

```
プロンプト設計原則:
「〜以内」→ 怠けた出力を許す
「〜ぴったり」→ 推敲を強制する
```

### 1-2. 禁止ワードリスト — AI臭を消す最強の手法

```
禁止: 「しかし」「さらに」「つまり」「また」「一方で」「加えて」
許可: 「けど」「ただ」「要は」
```

この禁止リストがなぜ効くのか。LLMのデフォルトの接続詞選択は、訓練データ（ブログ記事、ニュース、学術論文）に引きずられる。「しかし」「さらに」「つまり」はまさにそのデフォルト語彙であり、**人間が読んだ瞬間に「AIが書いた文章だ」と判定するトリガーワード**になっている。

禁止することで、Claudeは口語寄りの接続詞（「けど」「ただ」「要は」）か、接続詞なしの短文連打を選ばざるを得なくなる。これがXの文体と完全に一致する。

```
プロンプト設計原則:
デフォルト語彙を禁止して、ターゲット語彙に強制的にシフトさせる。
「使ってほしい言葉」を指示するより「使うな」と言う方が効果が高い。
```

### 1-3. 「絵文字なし」の意図

X上の多くのバズアカウントは絵文字を多用する。あえて「絵文字なし」にすることで、タイムライン上での視覚的差別化が生まれる。テキストだけの投稿は「ガチ感」「硬派さ」を演出し、@oda_nobunaga10のブランドイメージ（武将キャラ・断言型）と一致する。

Claudeへの効果としても重要で、絵文字を許可すると文字数調整が複雑化し（絵文字は2文字分カウントされる場合がある等）、品質が不安定になる。

## 2. ペルソナ定義 — 「口調指定」の精度が異常に高い

### 2-1. 4つの口調マーカー

```
指定語彙: 「ガチ」「エグい」「〜だろ」「〜だよな」
```

この4語だけでペルソナが完全に固定される。なぜか。

- 「ガチ」→ 強調の入口。冒頭や要点に使う
- 「エグい」→ 感情の増幅。数値や事実のインパクトを倍増させる
- 「〜だろ」→ 読み手への問いかけ＆断定。距離を縮める
- 「〜だよな」→ 共感の誘導。読み手が「わかる」と感じるトリガー

この4語はそれぞれ**文章内での役割が異なる**。だからClaudeは自然に文章の各所にこれらを散らばせることができ、結果として全体のトーンが統一される。

もし口調指定が「カジュアルに書いて」だけだったら、Claudeは毎回違うカジュアルさで書いてしまう。**具体的な語彙を4つ限定指定する**ことで、ブレが消える。

```
プロンプト設計原則:
「〜な雰囲気で」→ 毎回ブレる
「この4語を使え」→ 毎回統一される
口調は形容詞ではなく語彙リストで定義する。
```

### 2-2. 「断定口調」の威力

```
禁止: 「〜かも」「〜と思う」「〜かもしれない」
強制: 「〜だ」「〜だろ」「〜だよな」「〜だぞ」
```

LLMのデフォルトはヘッジング（曖昧表現）方向に振れる。「〜かもしれません」「〜の可能性があります」を好む。これはXにおいて致命的に弱い。タイムラインを高速スクロールする人間の目に留まるのは、断定された強い文章だけだ。

断定口調の強制は、Claudeの「安全側に倒す本能」を上書きする。これだけで文章の説得力が一段上がる。

## 3. 構造テンプレート — 暗黙の3部構成

明文化はされていないが、今回の引用リポストには一貫した構造パターンが存在する。

```
[感情フック] → [事実の再解釈] → [行動喚起]
```

**実例（Viktor Oddy引用）:**
- 感情フック: 「ガチでやばい時代になった。」
- 事実の再解釈: 「Google StitchとClaudeでサイト作成比較してるけど、もうデザイナーに頼まなくてもプロ級のサイトが一瞬で出来る。」
- 行動喚起: 「道具は揃った。あとは行動するだけだよな。触らないやつは一生置いていかれる。今すぐ手を動かせ」

**実例（DeerFlow引用）:**
- 感情フック: 「これエグいだろ。」
- 事実の再解釈: 「ByteDanceがローカルで動くAIエージェントを無料で出してきた。データが外に出ないから安全で、研究から動画生成まで全部自分のPC上で完結する。」
- 行動喚起: 「ここで動く人と動かない人で圧倒的な差がつくんだよな。無料のうちに触っとけ。後から知らなかったは通用しない時代だぞ」

**実例（Claude Cowork引用）:**
- 感情フック: 「これガチで全員読め。」
- 事実の再解釈: 「Claude Coworkの活用事例20選を3万文字で無料公開とかエグすぎだろ。有料級の内容をX記事で出すこの姿勢がガチの信頼になるんだ。」
- 行動喚起: 「読んだだけで満足するやつが大半だけど、差がつくのは今日から1つでも実践した人間だけだよな。ブクマして終わりにするなよな」

3本とも同じ構造だが、読み手は「同じフォーマットだ」とは感じない。なぜなら、感情フックの語彙と事実の再解釈の角度が毎回変わるから。

```
プロンプト設計原則:
構造を固定し、表面を変える。
人間は構造のパターンには気づかないが、語彙のパターンには敏感。
だから構造はテンプレート化し、語彙だけバリエーションを持たせる。
```

## 4. 検証ループ — 機械的精度の担保

### 4-1. JavaScript文字数カウントの意味

```javascript
const text = '投稿文';
console.log([...text].length); // スプレッド構文でUnicode対応
```

このプロセスが毎回挟まることで、以下が担保される。

1. **全角/半角混在でも正確にカウントされる**（`String.length` ではなく `[...text].length` を使用）
2. **139でも141でもリジェクトされる** — 「だいたい140文字」は許容しない
3. **修正 → 再カウント → 確認のサイクルが高速で回る** — 人間がやると数分かかる推敲が数秒で完了する

この「文字数に対する完璧主義」が、投稿の密度と完成度に直結している。

### 4-2. 投稿後のスクリーンショット確認

投稿後に必ずプロフィールページに遷移してスクリーンショットを撮影し、投稿が正しく表示されているかを目視確認している。これにより「投稿したつもりが失敗していた」というリスクがゼロになる。

## 5. 元ポスト選定基準 — 露出とブランドの同時最適化

```
条件:
- いいね100以上
- AI/テック/ビジネス関連
- 直近数時間以内
```

この3条件は以下を同時に達成する。

| 条件 | 達成する効果 |
|---|---|
| いいね100以上 | そのポストを見ている人数が多い → 引用リポストのインプレッション底上げ |
| AI/テック/ビジネス関連 | ターゲット層（AI活用に興味がある層）への正確なリーチ |
| 直近数時間以内 | タイムラインの鮮度アルゴリズムに乗る → 表示されやすくなる |

重要なのは「バズってるポストに乗る」だけではなく、**ジャンルを絞っている**こと。美容やエンタメのバズポストに引用リポストしても、フォロワーの属性がズレる。ターゲットジャンルを限定することで、引用リポスト経由で流入するフォロワーの質が担保される。

## 6. Claudeの特性との相性 — なぜClaude × この制約が最適なのか

### 6-1. Claudeの長文要約能力

元ポストの内容を読み取り、140文字に圧縮する作業は、本質的には「超高圧縮の要約」だ。Claudeは長文を短文に凝縮する能力が特に高く、この制約との相性が良い。要約時に情報の優先度を正確に判断し、最も重要なポイントだけを残す。

### 6-2. 文字数の精密制御

「あと3文字足りない」「2文字多い」という調整作業において、Claudeは日本語の語彙の中から文字数が合う同義語を高速で探索できる。「行動する」→「行動」（2文字削減）、「ツール」→「このツール」（2文字追加）といった微調整を意味を崩さずに実行できる。

### 6-3. ペルソナの一貫保持

会話コンテキスト内で一度ペルソナが定義されると、Claudeはその口調を最後まで維持する。セッション内でブレない。これはGPT系と比較したときのClaudeの強みの一つで、長い会話でもキャラクターが崩れにくい。

## 7. 改善余地のある点

品質は高いが、更に改善できるポイントもある。

### 7-1. 構造テンプレートの明文化

現在、[感情フック → 事実の再解釈 → 行動喚起] の3部構成は暗黙知になっている。これをプロンプト内で明文化すれば、品質の安定度がさらに上がる。加えて、別の構造パターン（[疑問提示 → 反論 → 結論] など）をバリエーションとして持てば、フォロワーの飽きを防げる。

### 7-2. 引用文の差別化チェック

直近の投稿と構造や語彙が被っていないかをチェックする仕組みがない。「ガチで」「エグい」の使用頻度が偏ると、フォロワーがパターンに気づく。過去N件の投稿と照合して語彙の重複を避けるロジックがあるとさらに良い。

### 7-3. 「で、」問題の汎用化

今回「で、」を使わないようフィードバックをもらった。こうした「使ってみて初めてわかるNG表現」を禁止リストに随時追加していく仕組み（フィードバックループ）があると、運用を重ねるほど品質が上がり続ける。

## まとめ — プロンプト設計7原則（この運用から抽出）

| # | 原則 | 具体例 |
|---|---|---|
| 1 | 自由度を下げれば品質が上がる | 「140文字ぴったり」で推敲を強制 |
| 2 | 禁止リストは許可リストより強力 | 「しかし」を禁止 → 口語接続詞に自動シフト |
| 3 | 口調は語彙リストで定義する | 「カジュアルに」ではなく「ガチ」「エグい」「〜だろ」「〜だよな」 |
| 4 | 構造を固定し表面を変える | [感情フック → 再解釈 → 行動喚起] の3部構成 |
| 5 | 機械的検証を組み込む | JavaScriptで文字数を毎回検証 |
| 6 | 選定基準を数値化する | 「いいね100以上」「直近数時間」「AI/テック系」 |
| 7 | フィードバックを禁止リストに変換する | 「で、」を使わない → 禁止ワードに追加 |

この7原則は、X運用に限らずClaude活用全般に転用できる。自由に書かせないこと、検証を自動化すること、フィードバックを構造化すること。この3つが、LLMの出力品質を人間の期待値以上に引き上げる鍵になっている。
